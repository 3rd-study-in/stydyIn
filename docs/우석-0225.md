# DropdownSelect Atom 컴포넌트 제작 (2026-02-25)

## 만든 것

`src/atoms/DropdownSelect/` 폴더 아래 Figma 디자인 스펙을 기반으로 드롭다운/셀렉트 관련 UI 컴포넌트를 제작했다.

```
src/atoms/DropdownSelect/
├── _shared.jsx        공통 SVG 아이콘 + OptionList + useDropdown 훅
├── Dropdown.jsx       일반 드롭다운 (텍스트 + 화살표)
├── FilterDropdown.jsx 필터 칩 드롭다운 (필터 아이콘 + 레이블 + 화살표)
├── Select.jsx         아이콘 셀렉트 (텍스트 + 고정 우측 아이콘 슬롯)
└── index.js           3개 컴포넌트 일괄 re-export
```

---

## 컴포넌트별 역할

### Dropdown

`Dropdown.png` / `Dropdown-ON.png` 기반. 폼에서 값을 선택할 때 쓰는 일반적인 셀렉트 박스다. 닫힌 상태에서는 ▼, 열린 상태에서는 ▲ 화살표가 표시된다.

### FilterDropdown

`filter-dropdown-OFF.png` / `filter-dropdown-ON.png` 기반. 검색/필터 UI에서 쓰는 칩(chip) 형태의 드롭다운이다. 좌측에 깔때기 아이콘이 고정으로 붙고, 그 옆에 레이블 텍스트가 표시된다.

### Select

`Select-box-icon.png` 기반. Dropdown과 구조는 같지만 우측 아이콘이 상태(열림/닫힘)와 무관하게 고정된다. 달력 아이콘 등 커스텀 SVG를 `icon` prop으로 꽂는 슬롯 용도로 설계했다. SVG 파일이 아직 없어서 나중에 교체 예정이다.

---

## 설계 방식 선택 이유

### 1개 파일이 아닌 3개로 분리한 이유

처음에는 `variant` prop 하나로 분기하는 단일 파일(`DropdownSelect.jsx`)로 만들었다. 그런데 이 방식은 파일 하나가 여러 UI 형태를 동시에 책임지기 때문에, 한 variant를 수정할 때 다른 variant가 영향을 받는지 확인해야 하는 부담이 있다.

Figma 디자인에서도 이미지가 3종 세트로 나뉘어 있었기 때문에, 디자인 단위와 코드 단위를 맞추는 것이 다른팀원들이 확인할때 직관적이라고 판단해 3개로 분리했다.

### `_shared.jsx`를 둔 이유

3개 컴포넌트가 공통으로 사용하는 코드(SVG 아이콘, 옵션 리스트, open/close 훅)를 각 파일에 복사하면 수정 시 3곳을 모두 바꿔야 한다. `_shared.jsx`로 추출하면 한 곳만 수정하면 된다. `_` 접두사는 이 파일이 폴더 내부 전용이며 외부로 export되지 않는다는 관례적 표시다.

### SVG를 인라인으로 임시 구현한 이유

SVG 파일이 아직 없는 상태에서 Figma CSS의 `%` 좌표값을 `px`로 환산해 `<polygon>` / `<path>`로 직접 구현했다. 나중에 SVG 파일이 생기면 해당 컴포넌트의 `icon` prop에 꽂거나 `_shared.jsx`의 아이콘 함수만 교체하면 되도록 구조를 잡았다.

### 색상·글꼴을 arbitrary value로 쓴 이유

`src/index.css`의 `@theme` 블록은 아직 확정되지 않은 상태라 건드리지 않았다. 지금은 Tailwind의 arbitrary value(`text-[#8D9299]`, `border-[#D9DBE0]` 등)로 Figma 원본값을 직접 박아두었다. 나중에 디자인 시스템 토큰이 확정되면 `@theme`에 변수로 올리고 클래스만 교체하면 된다.

---

## 이로 인한 이점

- **파일 단위 책임 분리** — 디자인 3종과 파일 3개가 1:1로 대응되어 어떤 파일을 수정해야 하는지 즉시 알 수 있다.
- **공통 로직 단일 관리** — 옵션 리스트 hover 스타일, 외부 클릭 닫기, Escape 처리 등을 `_shared.jsx` 한 곳에서 관리한다.
- **SVG 교체 용이성** — 아이콘 함수가 명시적으로 분리되어 있어 SVG 파일로 교체할 때 수정 범위가 명확하다.
- **색상 변경 추적 용이** — arbitrary value로 박혀 있어서 에디터에서 hex값 검색만 해도 어디를 고쳐야 하는지 바로 찾을 수 있다.
- **배럴 export** — `index.js`를 통해 외부에서는 폴더 경로 하나로 3개 컴포넌트를 모두 import할 수 있다.

# Modal 컴포넌트 제작 (2026-02-25)

## 만든 것

`public/` 폴더의 디자인 이미지 3종(`alert.png`, `User-Info.png`, `경고창-4.png`)을 기반으로 모달 컴포넌트를 제작했다. 베이스 래퍼는 `atoms`에, 복합 모달은 `shared/components`에 배치했다.

```
src/atoms/Modal/
└── Modal.jsx              베이스 래퍼 (오버레이 유무 + ESC / 외부클릭 닫기)

src/shared/components/Modal/
├── Alert.jsx              소형 확인 다이얼로그 (alert.png)
├── UserInfoModal.jsx      사용자 정보 시트 (User-Info.png)
├── ReportModal.jsx        신고 모달 껍데기 (경고창-4.png)
└── index.js               4개 컴포넌트 일괄 re-export
```

---

## 컴포넌트별 역할

### Modal

모든 모달의 베이스 래퍼. `hasOverlay` prop으로 어두운 배경 유무를 제어한다. ESC 키와 배경 클릭 시 `onClose`가 호출된다. 내부는 `children` 슬롯으로 비워 두었다.

### Alert

`alert.png` 기반. 400×160px 크기의 소형 확인 다이얼로그다. 상단 메시지 영역(`children`)과 하단 2버튼 바(좌 확인 / 우 취소, 각 50%)가 고정된 레이아웃이다.

### UserInfoModal

`User-Info.png` 기반. 390×694px. 좌상단 지역 태그, 우상단 신고 링크, 테두리를 살짝 벗어난 파란 X 닫기 버튼이 구조에 포함된다. 프로필 콘텐츠는 `children` 슬롯으로 외부에서 주입한다.

### ReportModal

`경고창-4.png` 기반. 390×430px. 상단 타이틀과 하단 2버튼 바(Alert와 동일 구조)가 고정되고, 가운데 선택 항목 영역은 `children` 슬롯이다.

---

## 설계 방식 선택 이유

### atoms와 shared를 나눈 이유

`Modal.jsx`는 오버레이와 닫기 동작만 담당하는 단순 래퍼라 atoms에 뒀다. `Alert`, `UserInfoModal`, `ReportModal`은 Modal을 조합한 복합 컴포넌트이므로 shared/components에 뒀다. README 아키텍처 기준과도 일치한다.

### 내부 콘텐츠를 children으로 비운 이유

라디오 목록, 프로필 데이터 등 실제 내용은 사용하는 feature마다 다르다. 겉 구조(크기, 테두리, 버튼 바, 닫기)만 컴포넌트로 고정하고 안쪽은 슬롯으로 열어두면, 같은 껍데기를 다른 콘텐츠와 함께 재사용할 수 있다.

### hasOverlay prop으로 두 버전을 제공한 이유

어두운 배경이 필요한 경우(전체 화면 집중)와 불필요한 경우(인라인 팝업 느낌)가 함께 존재한다. 별도 컴포넌트로 나누는 대신 prop 하나로 분기해 관리 포인트를 줄였다.

---

## 이로 인한 이점

- **레이어 분리** — 베이스(atoms)와 조합 컴포넌트(shared)가 명확히 나뉘어 교체·확장이 쉽다.
- **콘텐츠 독립** — children 슬롯 덕분에 같은 모달 껍데기를 여러 feature에서 다른 내용으로 쓸 수 있다.
- **닫기 일관성** — ESC·외부클릭·닫기버튼 모두 Modal 베이스 한 곳에서 처리해 각 컴포넌트가 중복 구현하지 않는다.
- **색상 변경 추적 용이** — arbitrary value로 박혀 있어 hex값 검색만으로 수정 위치를 찾을 수 있다.

## 사용 예시

```jsx
import { Alert, UserInfoModal, ReportModal } from '../shared/components/Modal'

// 확인 다이얼로그
<Alert
  isOpen={alertOpen}
  onClose={() => setAlertOpen(false)}
  onConfirm={handleConfirm}
  confirmText="생성하기"
  cancelText="취소"
>
  스터디를 생성하시겠습니까?
</Alert>

// 사용자 정보 시트
<UserInfoModal
  isOpen={userInfoOpen}
  onClose={() => setUserInfoOpen(false)}
  onReport={handleReport}
  location="애월읍"
>
  <ProfileContent user={user} />
</UserInfoModal>

// 신고 모달 (오버레이 없는 버전)
<ReportModal
  isOpen={reportOpen}
  onClose={() => setReportOpen(false)}
  onConfirm={handleSubmit}
  hasOverlay={false}
>
  <ReportOptions value={selected} onChange={setSelected} />
</ReportModal>
```
