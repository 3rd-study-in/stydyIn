## 1. Auth + Notification Zustand 스토어 구현

### 한 것21121

- `src/stores/authStore.js` — JWT 기반 인증 전역 상태
  - `decodeJwt()`, `isExpired()` 헬퍼로 토큰 만료 체크
  - `initAuth()`: 앱 시작 시 1회 호출, localStorage에서 토큰 복원
  - `setTokens()`: 로그인 성공 후 스토어 + localStorage에 토큰 저장
  - `logout()`: 스토어 + localStorage 전체 정리
- `src/stores/notificationStore.js` — 알림 전역 상태 + API 연동
  - `notifications`, `hasUnread`, `prevIds`(Set) 상태
  - `fetchNotifications()`: GET 후 이전 ID와 비교해 새 알림 감지
  - `markAllRead()`, `deleteOne(id)` 액션

### 왜 이렇게 했는가

- **Redux 대신 Zustand**: 보일러플레이트 없이 ~50줄로 전역 상태 관리 가능. actions/reducers 파일 분리 불필요.
- **`isLoggedIn` 단일 진실 공급원**: 컴포넌트마다 localStorage를 직접 읽지 않고 스토어 하나로 통일
- **`initAuth()`를 `main.jsx`에서 최초 1회 호출**: 새로고침해도 로그인 상태 복원, JWT exp 체크로 만료된 토큰 자동 정리
- **`prevIds` Set 패턴**: 이전 알림 ID 목록과 현재를 비교해 `hasUnread` 감지 → 매번 서버에 읽음 여부 묻지 않아도 됨

## 8. 라우터 선택 및 라우팅 구조 설계

### 한 것

React Router v7을 채택하고, Nested Layout 방식의 라우팅 구조를 설계했다.
내일 바로 `src/App.jsx`에 적용할 수 있도록 `docs/routing-plan.md`에 실행 계획을 문서화했다.

### 라우터 선택: React Router v7

| 라우터          | 특징                                                               |
| --------------- | ------------------------------------------------------------------ |
| React Router v7 | 이미 설치됨, 가장 큰 생태계, loader/action 데이터 페칭 통합        |
| TanStack Router | 파일 기반 라우팅, 완전한 타입 안전성 (TypeScript 없으면 장점 반감) |
| Wouter          | ~2KB 초경량, 단순한 API (고급 기능 없음)                           |

**왜 React Router v7인가:**

- 이미 설치되어 있음 (`react-router-dom@7.13.1`) → 추가 작업 없이 바로 사용 가능
- 프로젝트가 JSX 기반이라 TypeScript 전용 장점인 TanStack Router의 타입 추론이 불필요
- JWT 인증 + 준회원/정회원 분기 같은 **가드 라우팅**을 레이아웃 컴포넌트로 깔끔하게 처리 가능
- Wouter는 중첩 레이아웃, 가드 같은 기능이 없어 이 프로젝트 요구사항을 충족하지 못함

### 라우팅 구조 탐색: 3가지 패턴 비교

**패턴 1 — Flat (단순 나열형)**

모든 라우트에 Header, Footer를 직접 JSX로 감싸는 방식.

```jsx
{ path: '/login', element: <><GNBLogin /><LoginPage /><Footer /></> }
{ path: '/', element: <><GNB /><HomePage /><Footer /></> }
// 라우트마다 Header/Footer 반복
```

- 장점: 구조 파악이 직관적
- 단점: Header/Footer 코드 중복, 라우트 추가마다 복붙 필요

**패턴 2 — Nested Layout (중첩 레이아웃)**

레이아웃 컴포넌트를 부모 라우트로 두고, 자식 페이지는 `<Outlet />`으로 렌더링.

```jsx
{ element: <PublicLayout />, children: [{ path: '/login', element: <LoginPage /> }] }
{ element: <PrivateLayout />, children: [{ path: '/', element: <HomePage /> }] }
```

- 장점: Header/Footer 중복 제거, 인증 가드를 레이아웃 한 곳에서 처리
- 단점: `<Outlet />` 개념이 처음엔 낯설 수 있음

**패턴 3 — Routes 분리 파일**

라우트 설정을 `src/routes/index.jsx` 별도 파일로 분리.

- 장점: App.jsx가 단순해짐, 라우트 설정만 따로 관리
- 단점: 7개 페이지 수준에서 파일 분리는 **과한 분리**, 참조할 파일이 늘어남

### 채택: Nested Layout (패턴 2)

**채택 이유 4가지:**

1. **GNB / GNBLogin 두 가지 헤더 존재** — 로그인 여부에 따라 헤더가 달라짐. 레이아웃 분리로 조건 분기 없이 처리
2. **인증 가드 일원화** — 보호 라우트마다 개별 인증 체크 없이 `PrivateLayout` 하나에서 처리
3. **Header / Footer 중복 제거** — Flat 방식 대비 반복 코드 없음
4. **프로젝트 규모에 적합** — 총 8개 페이지 수준에서 routes.jsx 별도 분리는 과한 분리

### 최종 레이아웃 구조

| 레이아웃        | 헤더                   | 인증 가드             | 해당 페이지                 |
| --------------- | ---------------------- | --------------------- | --------------------------- |
| `AuthLayout`    | `GNBLogin`             | X                     | /login, /signup             |
| `GeneralLayout` | `GNB` (내부 자동 분기) | X                     | /, /search                  |
| `PrivateLayout` | `GNB`                  | O → /login 리다이렉트 | /study/\*, /profile/:userId |
